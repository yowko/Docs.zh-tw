# <a name="peer-channel-security"></a><span data-ttu-id="8d2b9-102">對等通道安全性</span><span class="sxs-lookup"><span data-stu-id="8d2b9-102">Peer Channel Security</span></span>
<span data-ttu-id="8d2b9-103">「對等通道」能夠啟用各種相依於多方通訊的分散式應用程式類型。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-103">Peer Channel enables a variety of distributed application types that depend on multiparty messaging.</span></span> <span data-ttu-id="8d2b9-104">有些範例會包含網際網路範圍的內容散發，其中受信任的來源會散發內容 (例如媒體或軟體更新)、朋友們會互相交換音樂或相片，或是同事們一起編輯文件。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-104">Some examples include Internet-scale content distribution, where a trusted source distributes content (such as media or software updates), a group of friends exchange music and photos, or a team of colleagues collaboratively edit a document.</span></span> <span data-ttu-id="8d2b9-105">每個案例都需要唯一的安全性模型。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-105">Each of these scenarios requires a unique security model.</span></span> <span data-ttu-id="8d2b9-106">「對等通道」安全性模型的設計目的，是處理這些案例並提供聲音安全性模型，以滿足不同識別、驗證和授權模型的個別需求。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-106">The Peer Channel security model is designed to address these scenarios and provides a sound security model for the respective needs of different identity, authentication, and authorization models.</span></span>  
  
## <a name="security-scenarios"></a><span data-ttu-id="8d2b9-107">安全性案例</span><span class="sxs-lookup"><span data-stu-id="8d2b9-107">Security Scenarios</span></span>  
 <span data-ttu-id="8d2b9-108">內容散發案例需要每個內容收件者識別內容來源。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-108">A content-distribution scenario requires that each content recipient identify the content source.</span></span> <span data-ttu-id="8d2b9-109">基於案例的分散式特性，不太可能永遠得知與信任處理或攔截訊息的媒介。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-109">Due to the distributed nature of the scenario, it is not always possible to know and trust the intermediaries that process or intercept messages.</span></span> <span data-ttu-id="8d2b9-110">若要有效降低不受信任媒體可能會竄改這些訊息的威脅，應用程式可以在傳送者端保護訊息的安全，藉此輕易地偵測出任何嘗試竄改的威脅。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-110">To effectively mitigate the threat that an untrusted intermediary might tamper with the messages, applications can secure the message at the sender so that any tampering attempts are easily detected.</span></span> <span data-ttu-id="8d2b9-111">在此例中，依據內容的機密性可能會需要加密。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-111">In this case, depending on the confidentiality of the content, encryption can be necessary.</span></span>  
  
 <span data-ttu-id="8d2b9-112">共同作業案例 (像是群組文件共同作業) 通常會需要每個成員經過個別識別與驗證才能參與工作階段。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-112">Collaboration scenarios like group document collaboration often require that each member participating in the session be individually identified and authenticated.</span></span> <span data-ttu-id="8d2b9-113">這表示需要有定義使用者群組和驗證這些群組的機制，才能保護工作階段的安全。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-113">This means that a mechanism to define user groups and authenticate against those groups is necessary to have a secured session.</span></span> <span data-ttu-id="8d2b9-114">此外，應用程式可能需要在訊息層級進行驗證以追蹤每個訊息。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-114">Moreover, applications might require tracing each message by authentication at the message level.</span></span> <span data-ttu-id="8d2b9-115">在這些應用程式類型中，較嚴密的安全性配置可能會犧牲效能。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-115">In these types of applications, performance can be sacrificed for a stronger security scheme.</span></span>  
  
 <span data-ttu-id="8d2b9-116">非必要之使用者群組間的通訊工作階段，可能會需要非正式的安全性模型，就像是群組之間共通秘密的知識。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-116">A communication session among a group of casual users can require informal security models, like knowledge of a common secret among the group.</span></span> <span data-ttu-id="8d2b9-117">就這些應用程式類型而言，必須要能夠輕鬆的建立與設定安全性模型，且相形之下，建立最嚴密的驗證形式或實施不可否認性的預防，則重要性較低。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-117">For these types of applications, having a security model that is convenient to establish and configure is more important than having the strongest form of authentication or providing nonrepudiation measures.</span></span> <span data-ttu-id="8d2b9-118">針對這些案例，以密碼為基礎的驗證機制能夠協助保護通訊層的安全性，同時仍然允許進行訊息驗證。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-118">For these scenarios, a password-based authentication mechanism helps to secure the communication layer while still allowing for message authentication.</span></span> <span data-ttu-id="8d2b9-119">以密碼為基礎的安全性是對等通道的預設設定。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-119">Password-based security is the default setting for Peer Channel.</span></span>  
  
## <a name="token-types"></a><span data-ttu-id="8d2b9-120">語彙基元類型</span><span class="sxs-lookup"><span data-stu-id="8d2b9-120">Token Types</span></span>  
 <span data-ttu-id="8d2b9-121">對等通道會辨認嚴密識別 (X.509 憑證) 的單一權杖類型，它會根據能夠實作的驗證與授權類型，提供嚴密的識別模型。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-121">Peer Channel recognizes a single token type for strong identification, X.509 certificates, which provide a strong identity model based on the type of authentication and authorization that can be implemented.</span></span> <span data-ttu-id="8d2b9-122">使用憑證能夠輕鬆地提供機密性和完整性。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-122">Confidentiality and integrity are easily provided using certificates.</span></span> <span data-ttu-id="8d2b9-123">但是，X.509 憑證可能會很難使用與部署。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-123">However, X.509 certificates can be difficult to use and deploy.</span></span>  
  
 <span data-ttu-id="8d2b9-124">對等通道也會透過使用密碼的方式提供簡單應用程式的支援。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-124">Peer Channel also provides support for simple applications through the use of passwords.</span></span> <span data-ttu-id="8d2b9-125">應用程式可以選擇根據所提供的密碼，設定快速而簡單的對等群組。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-125">Applications can choose to set up quick and simple peer groups based on a supplied password.</span></span> <span data-ttu-id="8d2b9-126">在此例中，群組擁有者會決定密碼，並與成員互相通訊密碼。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-126">In this case, a group owner decides and communicates the password to members.</span></span> <span data-ttu-id="8d2b9-127">每個成員必須使用這個密碼進行登入，才能夠加入工作階段。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-127">Each member must sign in using this password before they can join the session.</span></span> <span data-ttu-id="8d2b9-128">密碼只能用來允許進入工作階段；而不能用來執行訊息驗證。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-128">Passwords can be used only to allow entry to the session; they cannot be used to perform message authentication.</span></span> <span data-ttu-id="8d2b9-129">這是因為對等群組共享的對稱式權杖，如果使用在來源驗證方面會發生困難而且不適當。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-129">This is because a symmetric token that a group of peers share is difficult and inappropriate to use for source authentication.</span></span>  
  
## <a name="security-model"></a><span data-ttu-id="8d2b9-130">安全性模型</span><span class="sxs-lookup"><span data-stu-id="8d2b9-130">Security Model</span></span>  
 <span data-ttu-id="8d2b9-131">對等通道提供保護對等之間個別連結安全性的功能。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-131">Peer Channel provides the ability to secure the individual links between peers.</span></span> <span data-ttu-id="8d2b9-132">這表示訊息永遠不會在不安全的連結上流動 (從應用程式的觀點來看)。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-132">This means that a message never flows on an unsecured link (from the application perspective).</span></span> <span data-ttu-id="8d2b9-133">對內部來說，會使用傳輸層安全性 (TLS) 保護每個連結 (兩個對等之間的傳輸通道) 的安全。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-133">Internally, each link (a transport channel between two peers) is secured using Transport Layer Security (TLS).</span></span> <span data-ttu-id="8d2b9-134">這表示當傳送者撰寫並傳送訊息時，會透過安全的傳輸傳送到每個直接對等，而對等就可以存取訊息，然後再透過安全連線將訊息傳送到直接對等。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-134">This means that when a sender composes and sends a message, it is sent over secure transport to each of its immediate peers, who access the message, and in turn send the message to their immediate peers over secure connections.</span></span> <span data-ttu-id="8d2b9-135">這個安全性只有在傳輸層才有效，並且與訊息安全性模型無關。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-135">This security only works at the transport level and is independent of the message security models.</span></span>  
  
 <span data-ttu-id="8d2b9-136">無論使用了哪種傳輸安全性，對等通道也一定會提供保護訊息安全性的方法。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-136">Peer Channel also provides a way to secure messages independently of the transport security used.</span></span> <span data-ttu-id="8d2b9-137">在這個模型中，會使用來源的安全性權杖在來源處保護訊息的安全性 (雖然目前只支援 X.509 憑證)。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-137">In this model, the message is secured at the source using the source’s security token, although currently only X.509 certificates are supported.</span></span> <span data-ttu-id="8d2b9-138">然後透過對等網路傳輸受保護的訊息。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-138">The secured message is then transmitted over the peer network.</span></span> <span data-ttu-id="8d2b9-139">每個接收對等可以驗證來源的真實性。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-139">Each receiving peer can verify the authenticity of the source.</span></span> <span data-ttu-id="8d2b9-140">請注意訊息已經受到保護，因此媒介就無法竄改它。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-140">Note that the message is secured so that intermediaries cannot tamper with it.</span></span>  
  
 <span data-ttu-id="8d2b9-141">若要達到機密性，應用程式可以搭配強式群組成員資格配置採用傳輸安全性，以避免訊息受到未經授權的存取。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-141">To achieve confidentiality, applications can employ transport security with strong group membership schemes to prevent unauthorized access to the message.</span></span>  
  
 <span data-ttu-id="8d2b9-142">只要應用程式選擇其中一個支援的權杖類型，對等通道就不需要特定的識別模型。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-142">Peer Channel does not require a specific identity model as long as the application chooses one of the supported token types.</span></span> <span data-ttu-id="8d2b9-143">應用程式完全擁有這些識別和驗證決策的生命週期。</span><span class="sxs-lookup"><span data-stu-id="8d2b9-143">Applications completely own the life cycle of these identities and authentication decisions.</span></span>
