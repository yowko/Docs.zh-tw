# <a name="what-is-managed-code"></a><span data-ttu-id="23858-103">什麼是 Managed 程式碼？</span><span class="sxs-lookup"><span data-stu-id="23858-103">What is "managed code"?</span></span>

<span data-ttu-id="23858-104">使用 .NET Framework 時，您經常會遇到「Managed 程式碼」一詞。</span><span class="sxs-lookup"><span data-stu-id="23858-104">When working with .NET Framework, you will often encounter the term "managed code".</span></span> <span data-ttu-id="23858-105">本文件將說明此詞彙所代表的意義及其他相關資訊。</span><span class="sxs-lookup"><span data-stu-id="23858-105">This document will explain what this term means and additional information around it.</span></span>

<span data-ttu-id="23858-106">簡單來說，Managed 程式碼就是其執行受到執行階段管理的程式碼。</span><span class="sxs-lookup"><span data-stu-id="23858-106">To put it very simply, managed code is just that: code whose execution is managed by a runtime.</span></span> <span data-ttu-id="23858-107">在此情況下，不論實作為何 ([Mono](https://www.mono-project.com/)、.NET Framework 或 .NET Core)，所提到的執行階段都稱為 **Common Language Runtime** 或 CLR。</span><span class="sxs-lookup"><span data-stu-id="23858-107">In this case, the runtime in question is called the **Common Language Runtime** or CLR, regardless of the implementation ([Mono](https://www.mono-project.com/) or .NET Framework or .NET Core).</span></span> <span data-ttu-id="23858-108">CLR 負責將 Managed 程式碼編譯成機器碼，再加以執行。</span><span class="sxs-lookup"><span data-stu-id="23858-108">CLR is in charge of taking the managed code, compiling it into machine code and then executing it.</span></span> <span data-ttu-id="23858-109">此外，執行階段提供幾項重要服務，例如自動記憶體管理、安全性界限、型別安全等。</span><span class="sxs-lookup"><span data-stu-id="23858-109">On top of that, runtime provides several important services such as automatic memory management, security boundaries, type safety etc.</span></span>

<span data-ttu-id="23858-110">與此相反的是您執行 C/C++ 程式的方式，又稱為「Unmanaged 程式碼」。</span><span class="sxs-lookup"><span data-stu-id="23858-110">Contrast this to the way you would run a C/C++ program, also called "unmanaged code".</span></span> <span data-ttu-id="23858-111">在 Unmanaged 世界中，程式設計人員會負責處理幾乎所有工作。</span><span class="sxs-lookup"><span data-stu-id="23858-111">In the unmanaged world, the programmer is in charge of pretty much everything.</span></span> <span data-ttu-id="23858-112">實際程式基本上是作業系統 (OS) 載入記憶體並啟動的二進位檔。</span><span class="sxs-lookup"><span data-stu-id="23858-112">The actual program is, essentially, a binary that the operating system (OS) loads into memory and starts.</span></span> <span data-ttu-id="23858-113">從記憶體管理到安全性考量等其他工作都是程式設計人員的責任。</span><span class="sxs-lookup"><span data-stu-id="23858-113">Everything else, from memory management to security considerations are a burden of the programmer.</span></span>

<span data-ttu-id="23858-114">Managed 程式碼是以其中一種高階語言撰寫而成 (例如 C#、Visual Basic、F# 等)，並可在 .NET 上執行。</span><span class="sxs-lookup"><span data-stu-id="23858-114">Managed code is written in one of the high-level languages that can be run on top of .NET, such as C#, Visual Basic, F# and others.</span></span> <span data-ttu-id="23858-115">當您使用這些語言各自的編譯器來編譯以這些語言撰寫的程式碼時，您不會取得機器碼。</span><span class="sxs-lookup"><span data-stu-id="23858-115">When you compile code written in those languages with their respective compiler, you don’t get machine code.</span></span> <span data-ttu-id="23858-116">您會取得稍後可供執行階段編譯及執行的**中繼語言**碼。</span><span class="sxs-lookup"><span data-stu-id="23858-116">You get **Intermediate Language** code which the runtime then compiles and executes.</span></span> <span data-ttu-id="23858-117">這項規則的唯一例外是 C++，因為它也會產生在 Windows 上執行的原生、Unmanaged 二進位檔。</span><span class="sxs-lookup"><span data-stu-id="23858-117">C++ is the one exception to this rule, as it can also produce native, unmanaged binaries that run on Windows.</span></span>

## <a name="intermediate-language--execution"></a><span data-ttu-id="23858-118">中繼語言和執行</span><span class="sxs-lookup"><span data-stu-id="23858-118">Intermediate Language & execution</span></span>

<span data-ttu-id="23858-119">什麼是「中繼語言」(簡稱 IL)？</span><span class="sxs-lookup"><span data-stu-id="23858-119">What is "Intermediate Language" (or IL for short)?</span></span> <span data-ttu-id="23858-120">它是以高階 .NET 語言撰寫之程式碼的編譯結果。</span><span class="sxs-lookup"><span data-stu-id="23858-120">It is a product of compilation of code written in high-level .NET languages.</span></span> <span data-ttu-id="23858-121">當您編譯以其中一種語言撰寫的程式碼之後，您會取得由 IL 所組成的二進位檔。</span><span class="sxs-lookup"><span data-stu-id="23858-121">Once you compile your code written in one of these languages, you will get a binary that is made out of IL.</span></span> <span data-ttu-id="23858-122">請務必注意，IL 與在執行階段上執行的任何特定語言都無關；它甚至有個別的規格，如果您想要，可以閱讀此規格。</span><span class="sxs-lookup"><span data-stu-id="23858-122">It is important to note that the IL is independent from any specific language that runs on top of the runtime; there is even a separate specification for it that you can read if you’re so inclined.</span></span>

<span data-ttu-id="23858-123">當您從高階程式碼產生 IL 之後，您可能想要執行它。</span><span class="sxs-lookup"><span data-stu-id="23858-123">Once you produce IL from your high-level code, you will most likely want to run it.</span></span> <span data-ttu-id="23858-124">此時 CLR 會接管，並開始 **Just-In-Time** 編譯 (或 **JIT-ing**) 的程序，以將您的程式碼從 IL 編譯成可實際在 CPU 上執行的機器碼。</span><span class="sxs-lookup"><span data-stu-id="23858-124">This is where the CLR takes over and starts the process of **Just-In-Time** compiling, or **JIT-ing** your code from IL to machine code that can actually be run on a CPU.</span></span> <span data-ttu-id="23858-125">如此一來，CLR 就會知道您程式碼的實際功能，並可有效地進行「管理」。</span><span class="sxs-lookup"><span data-stu-id="23858-125">In this way, the CLR knows exactly what your code is doing and can effectively _manage_ it.</span></span>

<span data-ttu-id="23858-126">中繼語言有時也稱為通用中間語言 (CIL) 或 Microsoft Intermediate Language (MSIL)。</span><span class="sxs-lookup"><span data-stu-id="23858-126">Intermediate Language is sometimes also called Common Intermediate Language (CIL) or Microsoft Intermediate Language (MSIL).</span></span>

## <a name="unmanaged-code-interoperability"></a><span data-ttu-id="23858-127">Unmanaged 程式碼互通性</span><span class="sxs-lookup"><span data-stu-id="23858-127">Unmanaged code interoperability</span></span>

<span data-ttu-id="23858-128">當然，CLR 允許超過 Managed 與 Unmanaged 世界之間的界限，而且有許多程式碼會這樣做，甚至是在[基底類別庫](framework-libraries.md)中。</span><span class="sxs-lookup"><span data-stu-id="23858-128">Of course, the CLR allows passing the boundaries between managed and unmanaged world, and there is a lot of code that does that, even in the [Base Class Libraries](framework-libraries.md).</span></span> <span data-ttu-id="23858-129">這稱為**互通性** 或簡稱 **Interop**。</span><span class="sxs-lookup"><span data-stu-id="23858-129">This is called **interoperability** or just **interop** for short.</span></span> <span data-ttu-id="23858-130">這些佈建可讓您包裝並呼叫 Unmanaged 程式庫。</span><span class="sxs-lookup"><span data-stu-id="23858-130">These provisions would allow you to, for example, wrap up an unmanaged library and call into it.</span></span> <span data-ttu-id="23858-131">不過，請務必請注意，一旦您這樣做，當程式碼超過執行階段的界限時，執行的實際管理會再次交由 Unmanaged 程式碼，因此會受到相同的限制。</span><span class="sxs-lookup"><span data-stu-id="23858-131">However, it is important to note that once you do this, when the code passes the boundaries of the runtime, the actual management of the execution is again in the hand of unmanaged code, and thus falls under the same restrictions.</span></span>

<span data-ttu-id="23858-132">同樣地，C# 語言可讓您在程式碼中直接使用 Unmanaged 建構 (例如資料指標)，方法是利用所謂的 **unsafe 內容**，來指定程式碼片段的執行不受 CLR 管理。</span><span class="sxs-lookup"><span data-stu-id="23858-132">Similar to this, C# is one language that allows you to use unmanaged constructs such as pointers directly in code by utilizing what is known as **unsafe context** which designates a piece of code for which the execution is not managed by the CLR.</span></span>
